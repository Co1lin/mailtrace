Provider field guides

Copy-paste-friendly settings for the common cases. The plaintext SMTP password is stored in the local SQLite database — for shared SMTP services, use a scoped API key rather than your real password whenever possible.

★ Transactional relay (recommended): SendGrid, Resend, Postmark, Mailgun, AWS SES

Sign up for any of these — they all have free tiers that cover homelab volume — and use the SMTP credentials they hand you. The pattern is always:

Hoste.g. smtp.sendgrid.net · smtp.resend.com · smtp.postmarkapp.com
Port587 (STARTTLS) or 465 (TLS)
Usernameliteral string the provider tells you to use (often apikey)
Passwordthe API key the provider gave you
From addressany address on a domain you've verified with the provider (DKIM/SPF set up via their dashboard)

Why this is the default recommendation in 2026:

  • Way better deliverability than residential / consumer SMTP — recipients won't bin your scan digests.
  • No MFA / app-password / "did the provider disable basic auth this quarter?" dance.
  • Revocable per-app API keys; rotation is one-click in the dashboard.
  • Dedicated webhooks for bounces and complaints (MailTrace doesn't consume them yet, but they're there if you need to debug).
  • Free tiers (SendGrid 100/day, Resend 3000/mo, Postmark 100/mo at signup) easily cover even an active MailTrace user.

Gmail / Google Workspace

Hostsmtp.gmail.com
Port587
EncryptionSTARTTLS
Usernameyour full Gmail / Workspace address
Passworda 16-character app password (not your normal password)

Generate an app password at myaccount.google.com/apppasswords. 2-step verification on the account is required to create one. App passwords show up like abcd efgh ijkl mnop — paste them with or without spaces, both work.

Workspace tenants: your admin can disable app passwords / SMTP for the whole org. If you can't reach the apppasswords page or your test send hangs at AUTH, fall back to a transactional relay above.

⚠ Microsoft 365 / Office 365 (work or school account) — limited support

This path is mostly broken in 2026. Microsoft started disabling SMTP AUTH (incl. with app passwords) on M365 org tenants in 2024 and the rollout is now near-universal: app passwords no longer help. There is no per-mailbox toggle that overrides the tenant decision. The only way to send from an M365 mailbox over SMTP today is OAuth2 + the SMTP.Send Graph permission via an Azure AD app registration — which MailTrace doesn't yet implement, because the simpler answer is to use a transactional relay (above).

If you want delivery to look like it came from your M365 address, the practical workflow is:

  1. Sign up for a transactional relay (Resend, Postmark, etc.).
  2. In the relay's dashboard, verify your custom domain by adding the DKIM CNAMEs and SPF includes they tell you to add to your DNS.
  3. Once verified, set From address in MailTrace to you@yourcompany.com — mail goes out via the relay's IPs but DKIM-signed by your domain, so recipients see "from you@yourcompany.com" with valid DMARC.
  4. You can also set up a forwarding rule in M365 so replies to that address still land in your Outlook inbox.

If you really want to try M365 SMTP AUTH anyway (maybe your tenant is one of the holdouts where it still works), the legacy field set is:

Hostsmtp.office365.com
Port587 · Encryption: STARTTLS
Username / Fromyour full email address (M365 rejects mismatched From)
Passwordapp password — only works if your tenant still has SMTP AUTH enabled

How to confirm SMTP AUTH is off (the common case): a tenant admin can run in PowerShell Get-TransportConfig | fl SmtpClientAuthenticationDisabled; if True, the tenant has shut it off org-wide. Re-enabling at the mailbox level via Set-CASMailbox -SmtpClientAuthenticationDisabled $false only works if the tenant default is False or $null. If you're not the tenant admin, treat this path as unavailable.
